# Live Delhi Metro Map

Real-time visualization: a Leaflet map of Delhi showing all 9 real DMRC lines, with
trains animated moving between stations, live headcount totals per segment, and a
per-line show/hide selector — all driven straight from Kafka at runtime (not
batch/history).

## Architecture

```
Kafka topic                      server.py (FastAPI)                Browser
metro-camera-events       →  consume (group "metro-live-map-      →  Leaflet map:
  (raw, per-coach JSON)          viewer", latest offsets only)         - line polylines
                                per-train state, live per-segment      - station markers
                                headcount totals                       - animated trains
                              → WebSocket /ws, 1×/sec snapshot         - segment headcount
                                                                          labels
```

- **No Schema Registry needed to read the topic**: the producer's
  `JSONSerializer` frames each value as `[1 magic byte][4-byte schema id][JSON
  bytes]` — plain JSON is self-describing, so `server.py` just strips that
  5-byte header and `json.loads()`s the rest.
- **Per-train headcount** is summed live as each of the 8 coach messages for a
  departure arrives (no waiting on Flink's 1-minute window) — visible within
  ~1s of the first coach event.
- **Per-segment headcount** ("total headcount between each station, right
  now") is computed by grouping current in-memory train state by `(line,
  direction, current_station, next_station)` and summing — entirely in
  `server.py`, not backed by any Flink table.
- **Train animation**: the backend sends `received_at` (when it first saw this
  departure) with each snapshot; the browser interpolates the marker's
  position between `current_station` and `next_station` using elapsed time
  versus that line's average real hop duration (from `metro_network.py`'s
  real-distance-based calibration).
- **Visual polish**: each line has a thin white dashed overlay whose offset
  animates continuously (suggests current flowing along the track), train
  badges pulse with a soft colored glow, and the connection status dot
  pulses green while live / turns red on disconnect.
- **Line selector**: a checkbox per line (colored swatch + name, from
  `/api/lines`' `colors`) plus an "All lines" toggle in the side panel.
  Unchecking a line hides its polyline/stations/trains/segment-labels and
  excludes it from the onboard-count stats; a station shared by multiple
  lines (a real interchange) stays visible as long as *any* of its lines is
  still checked.
- **Surge highlight** (optional, needs `enable_surge_detection = true` in
  Terraform — see `../terraform/README.md`): a large pulsing red circle at any
  station Flink has flagged as a real, statistically significant surge (`>=
  1.5x` a trailing baseline, via `ML_DETECT_ANOMALIES`). `server.py` consumes
  `metro_station_surge_anomalies` directly (not the slower Bedrock
  recommendation table) so the highlight appears as soon as Flink detects it.
  That table is deliberately created with `'value.format' = 'json-registry'`
  (`../flink-sql/06_station_surge_anomalies.sql`), so it decodes with the
  exact same 5-byte-header trick as the raw topic above — no Schema Registry
  credentials needed for this either. Set `ENABLE_SURGE_HIGHLIGHTS=false` (or
  leave `enable_surge_detection = false` in Terraform, which sets this
  automatically) to disable; a surge stays highlighted for `SURGE_TTL_SECONDS`
  (6 minutes) after last being reconfirmed, then clears on its own.

## Station coordinates

`stations.json` (real lat/lng for all 223 unique stations across all 9 lines) is
generated by `build_stations.py` from `vendor/dmrc_network.csv` (real DMRC station
data, including per-station distance-from-line-start, from
[Vinith-J/Delhi-Metro-Network-Analysis](https://github.com/Vinith-J/Delhi-Metro-Network-Analysis)),
using the same name-cleanup/rename rules used to build `metro_network.py`'s station
lists in the first place (both are derived from this one CSV in the same pass, so they
stay consistent). Two stations (`Noida Electronic City`, a post-2019 Blue Line extension
past the CSV's last mapped stop, and `Shiv Vihar`, a real terminus whose CSV row is
individually wrong) needed a manually-looked-up coordinate; 14 more stations whose CSV
rows are wrong but *not* termini were automatically re-interpolated from their nearest
trusted neighbors instead -- see the "chaotic zigzags" postmortem below for how those
were found and why a simple lat/lng range check wasn't enough to catch them all. See
`build_stations.py` for the exact mapping, `MANUAL_COORDS` and
`DISCARD_BAD_CSV_COORDS` for the documented exceptions, and `validate()` for the
automatic check that now runs after every build. Re-run it any time
`metro_network.py`'s station lists change:

```bash
python3 build_stations.py
```

## Running it

**Recommended**: deploy everything (Confluent Cloud infra + this + the producer) with one
`terraform apply` — see [`../terraform/README.md`](../terraform/README.md). That builds
this directory into a container via `Dockerfile` (build context is the repo root, since
`server.py` reuses `../metro_network.py` at runtime) and runs it with real credentials
already wired in.

To run it directly instead:

```bash
cd live-map
pip install -r requirements.txt
uvicorn server:app --host 0.0.0.0 --port 8765
```

Then open `http://localhost:8765`. Make sure `python-producer.py` (in the
parent directory) is also running against the same Kafka cluster, or the map
will just show empty tracks with no trains.

Credentials are read from the existing `../.env` (same cluster/topic as the
producer — no new Confluent Cloud resources needed for this feature). If you
deployed via Terraform, copy `../terraform/generated.env` to `../.env` first.

## Postmortem: the marker-alignment bug

Trains briefly rendered scattered off the track entirely rather than following it. Root
cause: `.train-marker { position: relative; ... }` in `style.css` (added so the pulsing
glow `::after` pseudo-element would position relative to it) was overriding Leaflet's own
`.leaflet-marker-icon { position: absolute }` rule from `leaflet.css`. Since both rules
have equal CSS specificity and ours loaded later, `relative` won — which dropped every
marker div back into normal document flow (stacking down the page) instead of being
positioned purely by Leaflet's `transform: translate3d(...)`. The fix was simply not
setting `position` at all on `.train-marker`: Leaflet's own `position: absolute` already
provides everything the `::after` child needs.

Confirmed via a headless-browser check (Playwright) comparing each marker's actual
`getBoundingClientRect()` center against `map.latLngToContainerPoint()` for its intended
coordinate — 0px drift on all 38 active trains after the fix.

## Postmortem: containerized live-map crashing on `KeyError: 'SCHEMA_REGISTRY_URL'`

`server.py` used to reuse `YELLOW_LINE`/`BLUE_LINE`/`SEGMENT_TIMES` by loading
`../python-producer.py` as a module (`importlib`). That works locally by accident, but
executes *all* of `python-producer.py`'s top-level code as a side effect -- including
building a `SchemaRegistryClient` and a Kafka `Producer`, both of which read required env
vars at import time. In Docker, `docker_container.live_map` was only ever given Kafka
credentials (correctly -- this server never touches Schema Registry itself), so the import
crashed on start looking for `SCHEMA_REGISTRY_URL`. The real fix: the pure network/timing
model (station lists, segment travel times, route math -- no env vars, no Kafka clients)
now lives in its own module, `../metro_network.py`, which both `python-producer.py` and
`server.py` import directly. `server.py` no longer touches `python-producer.py` at all.
Verified by running the live-map container with *only* `BOOTSTRAP_SERVER`/`KAFKA_API_KEY`/
`KAFKA_API_SECRET` set (no Schema Registry vars whatsoever) and confirming it starts and
serves correctly.

## Postmortem: lines rendering as chaotic zigzags/"spider legs" instead of real tracks

After adding all 9 lines, several lines' polylines shot off in long straight spikes to
places their real tracks never go (e.g. the Violet Line appearing to detour out to
Gurugram, which it doesn't). Root cause: `vendor/dmrc_network.csv`, while structurally
reliable (verified earlier by confirming its distance-sorted order matches real DMRC
geography), has **15 individual rows with outright wrong coordinates** -- tens of km
from where those stations actually are, while their *distance-from-line-start* value
(used to derive segment timing) is correct. A simple lat/lng range check (Delhi/NCR's
real bounding box) had only caught one of these ("Shyam Park", whose row has the exact
same value copy-pasted into both its Latitude and Longitude columns) -- most of the
other 14 are individually *within* the Delhi/NCR range, just in the wrong *part* of it,
so a bounding-box check alone can't catch them.

The real fix, added to `build_stations.py`:
- `validate()` cross-checks each consecutive-station gap's straight-line distance
  against the CSV's own reliable distance-from-line-start delta for that same gap --
  real track curvature can make straight-line distance a bit *less* than track
  distance, but never dramatically more, so a large ratio reliably flags a bad
  coordinate without false-positiving on real long segments (e.g. Blue Line's Noida
  Electronic City extension).
- `DISCARD_BAD_CSV_COORDS` lists the 14 confirmed-bad, non-terminus stations found this
  way; they're treated as unresolved (same as a station missing from the CSV entirely),
  so they fall back to linear interpolation between their nearest trusted neighbors --
  fast and automatic, no manual per-station research needed. Two clusters shared
  suspiciously *identical* wrong coordinates across unrelated stations (one shared by
  "Old Faridabad" and all five "Noida Sector 34/52/61/59/62"; another shared by "Sarai"
  and "South Extension"), suggesting a geocoding fallback value in the source dataset
  rather than one-off typos.
- One station, "Shiv Vihar" (a real Pink Line *terminus*), couldn't be interpolated --
  no station beyond it to interpolate towards -- so it got a real, looked-up
  `MANUAL_COORDS` override instead, same treatment as "Shyam Park".

`resolve()` checks `MANUAL_COORDS` *before* the CSV lookup and `DISCARD_BAD_CSV_COORDS`
specifically so both mechanisms can override/reject known-bad rows, not just fill in
stations absent from the CSV entirely. `validate()` runs automatically at the end of
every `build_stations.py` run now, so a future CSV refresh introducing new bad rows
gets flagged immediately instead of requiring another by-eye map inspection.
